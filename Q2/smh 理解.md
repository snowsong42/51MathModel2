## 二、任务一：阶段转换节点识别（核心算法）

### 2.1 文档中设计的识别框架（3层过滤）

文档提出了一个 **3层递进的过滤机制**，从原始数据到最终节点：

```
原始速度 vi
    │
    ▼
第1层：稳健速度估计 ṽᵢ = median(vᵢ₋₂₁...vᵢ)    ← 窗口L=72点(12h)，消除噪声
    │
    ▼
第2层：瞬时跳变过滤                              ← 跳变指标 zᵢ > C_jump
    │   zᵢ = |vᵢ - ṽᵢ| / (MADᵢ + ε)              排除工程扰动引起的尖峰
    │   若 zᵢ > 5 → 标记为候选跳变点并跳过
    │
    ▼
第3层：真实阶段转换识别                          ← 阶跃量Rᵢ + 持久性检验
    │   ① 阶跃量 Rᵢ = (μ₂ - μ₁) / max(μ₁, δ)     速度水平是否显著上升(Rᵢ>0.6)
    │   ② 持久性 retreat(i) ≤ θ_back=0.5          6小时后是否不回落
    │   综合得分 Sᵢ = Rᵢ × 1{μ₂>μ₁} × 1{retreat≤0.5}
    │
    ▼
节点分类                                      ← 基于速度绝对水平区分
    节点1：匀速→加速 (μ_before≤0.6, μ_after≤5.0)
    节点2：加速→快速 (μ_after>5.0)
```

### 2.2 代码中的两套实现

有意思的是，两个主脚本采用了 **不同的实现策略**：

#### `stage_recognition.py` ✅ 完全对应文档算法

| 文档概念 | 代码实现 | 对应行 |
|---------|---------|-------|
| 稳健速度 `ṽᵢ` | `v_med[i] = np.median(v[i-L+1:i+1])` | 第23-26行 |
| MAD `MADᵢ` | `v_mad[i] = np.median(np.abs(win - v_med[i]))` | 第27行 |
| 跳变指标 `zᵢ` | `z_score = np.abs(v - v_med) / (v_mad + 1e-6)` | 第29行 |
| 阶跃量 `Rᵢ` | `R[i] = (mu2 - mu1) / mu1` | 第35-38行 |
| 持久性 `retreat(i)` | `retreat[i] = (mu2 - mu_future) / (mu2 - mu1 + 1e-6)` | 第40-43行 |
| 综合强度 `Sᵢ` | `S = R * (R > 0) * (retreat <= theta_back)` | 第45行 |
| 参数：`C_jump=5, C_trans=0.6, θ_back=0.5` | 完全一致 | 第17-19行 |

**但由于参数窗口 L=72(12h) 偏小，阈值设置可能导致节点识别不稳定，该脚本在运行时并未输出节点信息。**

#### `velocity.py` 🔄 简化替代方案

文档中的原算法在`stage_recognition.py`中实现后可能因为参数敏感度问题效果不佳，`velocity.py`采用了 **更简化的窗口滑动方法**：

```python
# 简化方案：直接看滑动窗口中位数速度是否超过阈值 + 持续性检验
thresh_slow = 0.15     # 速度 > 0.15 mm/h → 匀速→加速
thresh_fast = 2.0      # 速度 > 2.0 mm/h  → 加速→快速
L = 144                # 窗口 144点 (24小时)
```

**与文档参数对比**：
| 参数 | 文档推荐值 | velocity.py 实际值 | 差异原因 |
|-----|-----------|-------------------|---------|
| 低速阈值 | 0.6 mm/h | **0.15 mm/h** | 文档值偏大，实际数据匀速段速度更低 |
| 快速阈值 | 5.0 mm/h | **2.0 mm/h** | 文档值偏大，实际数据加速→快速在2mm/h左右 |
| 窗口长度 | 72点(12h) | **144点(24h)** | 拉长窗口提高中位数稳定性 |
| 节点间距 | 无要求 | **min_gap=500点** | 防止两个节点太近被误判 |

**运行结果验证**：
```
节点1 (匀速→加速): 序号 3074, 时间 21.34 天   ← 速度开始从~0.1上升
节点2 (加速→快速): 序号 8922, 时间 61.95 天   ← 速度突破2mm/h
```

这意味着：
- 前21天是缓慢匀速形变（平均速度 ~0.11 mm/h）
- 21~62天是加速形变（速度从0.11上升到~4.15 mm/h）
- 62天后进入快速形变（平均速度 ~4.15 mm/h）

## 三、任务二：阶段划分与建模（velocity.py核心实现）

### 3.1 阶段划分

基于识别出的节点序号，将数据切成三段：

| 阶段 | 序号范围 | 时间范围 | 数据量 |
|-----|---------|---------|-------|
| I - 匀速 | `1 : idx1` (1~3074) | 0 ~ 21.34天 | 512小时 |
| II - 加速 | `idx1 : idx2` (3074~8922) | 21.34~61.95天 | 974小时 |
| III - 快速 | `idx2 : end` (8922~10000) | 61.95~69.44天 | 179小时 |

### 3.2 各阶段建模（与文档完全一致）

**阶段 I - 线性模型**（匀速→位移线性增长）：
```python
d(t) = 0.108985 t - 7.76        # R²=0.9538, RMSE=3.55mm
```
- 斜率 = 平均速度 = **0.11 mm/h**

**阶段 II - 二次模型**（加速→位移抛物线增长）：
```python
d(t) = 3.29×10⁻⁴ t² - 0.243 t + 106.97   # R²=0.9871, RMSE=13.59mm
```
- 二次项系数正 → 加速度为正，速度持续增大
- 平均速度：**0.50 mm/h**（但末期速度更高）

**阶段 III - 线性模型**（快速→位移近乎直线陡升）：
```python
d(t) = 4.071511 t - 5594.57      # R²=0.9445, RMSE=51.19mm
```
- 斜率 = 平均速度 = **4.15 mm/h**（是阶段I的38倍！）

### 3.3 模型检验

文档要求R²和RMSE检验，代码中实现：
```python
r2 = r2_score(d, d_pred)                     # 决定系数
rmse = np.sqrt(mean_squared_error(d, d_pred)) # 均方根误差
```

| 指标 | 阶段I | 阶段II | 阶段III |
|-----|-------|-------|---------|
| **R²** | 0.9538 | **0.9871** | 0.9445 |
| **RMSE** | 3.55mm | 13.59mm | 51.19mm |

阶段II的R²最高(0.987)，说明二次模型很好地捕获了加速特征。
阶段III的RMSE最大(51mm)，因为该阶段数据点少但位移变化大(745mm)，线性近似较粗糙。

## 六、技术要点总结

1. **核心挑战**：从含噪声的时序数据中区分 "瞬时噪声跳变" 和 "真实阶段转换"
2. **文档方案**：3层过滤（稳健速度→MAD去跳变→阶跃量+持久性检验）
3. **实际方案**：简化版（速度水平滑动窗口+双阈值+持续性检验），参数根据数据实测调整
4. **建模选择**：匀速/快速用线性，加速用二次——符合物理直觉且简洁可解释
5. **代码与文档差异点**：`velocity.py`的阈值参数（0.15/2.0 vs 0.6/5.0）进行了调优，因为文档的推荐值偏保守；节点间距约束 `min_gap=500` 是文档未提及但实际必要的补充条件
